如果你过去两年一直在看 AI 编程产品的演示，可能已经很熟悉那套叙事了。

一句自然语言需求，模型吐出一段代码；或者一个报错扔进去，它帮你猜一个修法。这个阶段里，人们谈论“AI 会不会替代程序员”，通常讨论的是更接近应用层的工作：写页面、调接口、补测试、改业务逻辑。即便在最乐观的想象里，真正靠近硬件极限、需要长期反复试错的底层优化，似乎还是人类工程师的领域。

NVIDIA 这篇新论文最值得注意的地方，就在这里。

它研究的不是怎样让模型多写一点代码，而是怎样让一个 agent 连续工作 7 天，自主查文档、改实现、跑性能测试、修正确性问题，再继续优化，最后在 attention kernel 这种几乎已经被榨干的目标上，把性能做得比 cuDNN 和 FlashAttention-4 还高一点。

论文标题叫 A VO: Agentic Variation Operators for Autonomous Evolutionary Search。如果只看名字，它像一篇又一篇试图给 agent 架构起新概念的论文。但这篇文章真正有意思的，不是术语，而是它碰的对象足够难。

它选的是 attention。

这几乎是今天 AI 系统里最不缺优化的地方。Transformer 的核心运算就是 attention，而围绕 attention 做的内核优化，在过去几年里已经被 GPU 厂商、框架团队和论文作者反复打磨。FlashAttention 整个系列就是为了把这件事做得更快，NVIDIA 自己的 cuDNN 也在持续逼近硬件上限。换句话说，如果你想证明 agent 真能做出一些工程上有分量的事，拿 attention kernel 开刀，算是最不好挑软柿子的那种方式。

这篇论文的作者自己也很清楚这一点。文中反复强调，像这种已经被高度手工优化过的实现，难点不是“再生成一个候选代码版本”，而是要能持续和开发环境互动：去看硬件文档、去分析 profiler 输出、去发现瓶颈、去做修改、去跑测试、去处理错误、去根据前一次失败调整下一次策略。也就是说，真正稀缺的不是一次生成能力，而是反复迭代的能力。

这正是他们想把 agent 放进去的位置。

## 过去的 LLM，更像会提建议的人

论文里把过去这一类做法总结得很直接。像 FunSearch、AlphaEvolve 这一类“LLM 参与进化搜索”的方法，模型通常只负责一件事：根据已有候选解，生成一个新的候选。

剩下的事情，比如选哪个父代、怎么评估、哪些候选保留、怎样更新种群，通常由一个固定框架来决定。也就是说，大模型在这里更像一个会提建议的人。它可以给你一个新方案，但它并不真的掌握整个试错过程，也不能自己决定接下来该看什么资料、做什么测试、修什么问题。

NVIDIA 这篇论文觉得，这就是瓶颈。

因为对最难的优化问题来说，“生成一个候选”远远不够。真正的工程过程不是单轮问答，而是一种会不断打断自己、检查自己、修补自己、再继续向前推进的长循环。作者给这个新角色起名叫 AVO。名字可以先放一边，核心意思很简单：他们不再让大模型只负责吐出一个版本，而是让 agent 本身去扮演进化搜索里的“变异算子”。

这听起来有点抽象，但翻成更直白的话，就是：以前模型只是被叫来写一段代码，现在它要自己想办法把这段代码越改越好。

论文里的图把这件事画得很清楚。左边是传统做法：固定流程、单轮生成、外部框架主导；右边是 AVO：agent 在一个长循环里自己规划、实现、测试、调试，还能访问工具、记忆和过往版本。真正的变化不在“模型更大了”，而在它终于被允许进入工程循环本身。

## 结果不只是“还行”

这篇论文最容易被转发的当然是结果。

作者把 AVO 用在 NVIDIA Blackwell B200 GPU 上的 multi-head attention kernel 优化上，让它连续自主演化 7 天。最终得到的 kernel，在不同配置下最多可以达到 1668 TFLOPS，相对 cuDNN 最多提升 3.5%，相对 FlashAttention-4 最多提升 10.5%。如果再把已经演化出来的优化迁移到 grouped-query attention，也只额外花了 30 分钟自主适配，最后相对 cuDNN 最多提升 7.0%，相对 FlashAttention-4 最多提升 9.3%。

单看百分比，3.5% 这个数字在很多互联网产品环境里并不算惊人。但在这里，它的意义不太一样。

因为 attention kernel 已经不是一个“有很多低垂果实”的问题了。它本来就是整个 AI 软件栈里最被精打细磨的部分之一。FlashAttention-4 和 cuDNN 背后都是极强的工程团队，优化对象又高度贴近硬件上限。在这种地方还能再挤出几个百分点，难度和在一段普通 Python 业务代码上提速，根本不是一个量级。

论文里甚至专门强调，这不是一些表面的代码变形。作者分析 agent 找到的优化时说，这些修改涉及寄存器分配、指令流水调度、工作负载分配等多个层次。换句话说，它不是只会改变量名、重排 if-else，或者在模板上做一点简单调参，而是真的开始碰那些过去要靠非常资深的底层工程师才能摸到的东西。

如果这部分结果能被后续复现和扩展，它真正改写的，不会只是一个 benchmark 排名，而是大家对“AI 会先接管哪种工程工作”的判断。

## 重要的不是提速

这篇论文真正让我在意的地方，其实不是超过 FlashAttention-4 的那 10.5%。

而是它展示了一种已经很像工程劳动的 agent 形态。

过去很多“AI 编程”演示，其实都停留在比较短的回路里：你给一个需求，它给一个答案；答案不对，你再补一轮提示词。这个交互当然有价值，但它更接近问答系统，而不是工程系统。工程不是靠一次对话完成的，而是靠很多次“不对，再来一次”。

这篇论文里的 agent，被设计成会在循环中自己完成很多过去要靠人串起来的动作：它能访问以前的解、读领域知识库、决定先查什么、改哪里、什么时候调用评测工具、看到报错后怎么修、在停滞时是否换策略。它并不是一下子想出最优解，而是在一个长时间运行的环境里，不断积累局部改进。

这件事可能比“AI 会写代码”更重要。

因为一旦 AI 的能力中心从“生成”转向“迭代”，它能进入的工作就会不一样。很多真正有价值的软件和系统问题，不是缺一个聪明答案，而是缺一个愿意不断试错、不断修补、还能记住前面发生过什么的执行者。今天大家最焦虑的也许还是“模型能不能取代 junior 程序员”，但从这篇论文看，真正往前走的方向更可能是：它先开始承担那些高反馈密度、可评估、但非常耗费人类精力的循环性工程任务。

从这个角度看，论文标题里的 Autonomous Evolutionary Search 其实反而没那么重要。更值得记住的是另一层意思：AI 开始不满足于给建议，它想进入生产过程。

## attention 只是开始

当然，这类论文现在仍然有非常明显的边界。

首先，它的任务环境非常特殊。attention kernel 优化不是开放世界问题，而是一个有明确评测函数、明确目标硬件、明确性能指标的封闭工程环境。agent 可以不断试错，是因为它每次改动之后，都能很快得到反馈：快了还是慢了，对了还是错了。这种反馈密度，是很多现实工作并不具备的。

其次，这篇论文来自 NVIDIA，本身就拥有最好的硬件理解、最完备的内部工程环境，以及最适合拿来做这种实验的任务对象。它展示的是一种“在非常有利条件下，agent 能做到什么”，而不是“任何团队现在都能这么做”。

但即便把这些前提都放回去，这篇论文的信号还是非常明确。

它告诉你，AI 在工程里的下一步，可能不是更像 Copilot，而是更像一个不会下班的性能优化工程师。它不一定马上能独立做完所有事，但它已经开始出现在那些以前只有最硬核团队才会碰的工作区域里。

这也是为什么 attention 这个选题本身这么重要。它太难、太卷、太成熟，几乎不该成为“AI 再证明自己一次”的演示场景。可如果连这里都能被 agent 撬开一点，接下来更值得问的问题就不是“AI 会不会写代码”，而是：还有多少软件工程工作，本质上也只是一个长期迭代、可反馈、可验证的优化循环？

如果答案是“比大家原来以为的要多”，那这篇论文真正领先的地方，可能不是几个百分点的 TFLOPS，而是它把一条新分工线先画出来了。

过去，大模型负责回答问题；现在，它开始负责把问题做完。
